## 1. Intent

The **Proxy** pattern provides a surrogate or placeholder object that **controls access** to another object. The proxy and the real subject implement the same interface, so the client cannot tell the difference — it interacts with the proxy exactly as it would with the real object. The proxy intercepts calls and decides what to do: forward them immediately, defer them, cache results, check credentials, log activity, or talk to a remote process.

### Core Principle

> "Provide a substitute or placeholder for another object to control access to it."
> — Gang of Four, *Design Patterns: Elements of Reusable Object-Oriented Software*

The Proxy pattern is motivated by one fundamental reality: **direct access to an object is sometimes undesirable or impossible.** The object might be expensive to create, might live on a different machine, might require permissions to use, or might need observation for auditing purposes.

### Participants

| Participant       | Role                                                                    |
|-------------------|-------------------------------------------------------------------------|
| `Subject`         | Common interface shared by both RealSubject and Proxy                   |
| `RealSubject`     | The actual object that does real work; the proxy delegates to it        |
| `Proxy`           | Holds a reference to RealSubject; controls and augments access to it    |
| `Client`          | Talks to Subject interface; unaware whether it holds Proxy or Real      |

## 2. The Five Types of Proxies

### 2.1 Virtual Proxy

**Problem it solves:** Object creation is expensive (e.g., database fetch, large file load, heavy computation) and may not be needed at all.

**Mechanism:** The proxy creates the real object on first access (lazy initialization). Subsequent accesses reuse the already-created object. The client code does not change at all.

**Canonical examples:**
- Hibernate/JPA entity with `FetchType.LAZY` — the `@ManyToOne` collection is not loaded until you call the getter.
- A word processor that creates page objects only when the user scrolls to that page.
- Loading a 50 MB PDF thumbnail only when the user hovers over a document icon.

---

### 2.2 Protection Proxy

**Problem it solves:** Different clients should have different levels of access to the same object. You do not want to pollute the real object with authorization logic.

**Mechanism:** The proxy wraps the real object and checks permissions (role, scope, ownership) before delegating. Unauthorized calls throw a security exception or return gracefully without invoking the real subject.

**Canonical examples:**
- Spring Security's `@Secured`/`@PreAuthorize` interceptors.
- Unix file system permission checks before invoking `read()` / `write()`.
- An admin console where read-only users see the object but cannot call mutating methods.

---

### 2.3 Remote Proxy

**Problem it solves:** The real object lives in a different JVM, process, or machine. The client should not need to know about network transport.

**Mechanism:** The proxy handles all network serialization, connection management, and error handling. To the client it looks like a local call; the proxy translates it into a remote call (HTTP, gRPC, RMI, etc.).

**Canonical examples:**
- Java RMI stub (the generated stub is a remote proxy).
- gRPC client-side stub generated by `protoc`.
- Spring's `@FeignClient` or `RestTemplate`-based service clients.
- A distributed cache client (Jedis/Lettuce wrapping a Redis server).

---

### 2.4 Caching Proxy

**Problem it solves:** The real object's operation is expensive (network call, DB query, heavy computation) and the same result may be requested many times.

**Mechanism:** The proxy maintains a cache (in-memory Map, Redis, etc.) keyed on the arguments. On a cache hit it returns the cached result; on a miss it delegates to the real subject, stores the result, and returns it. A TTL (time-to-live) or eviction policy controls staleness.

**Canonical examples:**
- Spring `@Cacheable` — Spring wraps the bean in a proxy that intercepts the annotated method.
- HTTP reverse proxies (Nginx, Varnish, CloudFront) caching expensive origin responses.
- A DNS resolver caching domain-to-IP mappings with TTL.

---

### 2.5 Logging / Auditing Proxy

**Problem it solves:** Every call to a sensitive or business-critical operation must be recorded for compliance, debugging, or analytics, without tangling logging code into business logic.

**Mechanism:** The proxy intercepts each method call, records the caller identity, timestamp, method name, arguments, and return value (or exception), then delegates to the real subject.

**Canonical examples:**
- Spring AOP `@Around` advice logging service method entry/exit.
- Java's `java.lang.reflect.Proxy` wrapping a service interface to audit every call.
- Database statement interceptors (JDBC `PreparedStatement` wrappers) logging every SQL.

## 3. Real-World Analogy

### The Personal Assistant (Protection + Logging Proxy)

Imagine a busy CEO who receives hundreds of meeting requests, emails, and document-signing requests per day. Rather than letting everyone walk directly into the CEO's office, there is a **Personal Assistant (PA)**.

The PA is the **proxy**. The CEO is the **real subject**. Visitors are the **clients**.

- A board member arrives — the PA checks their identity and immediately escorts them in (**protection check passes, delegates**).
- An unknown cold-caller arrives — the PA blocks access (**protection check fails**).
- A document arrives for signature — the PA logs the receipt date, who sent it, and what it contains (**auditing**), then presents it to the CEO.
- The CEO asks the PA to collect the same market report every Monday — the PA caches last week's report if the request comes in before the new one is published (**caching**).
- The CEO hasn't arrived yet — the PA accepts work on their behalf until they show up (**virtual/lazy initialization**).

The visitor never knows whether they talked to the CEO directly or through the PA. The interface — "submit a request" — is identical.

## 4. When to Use / When NOT to Use

### Use the Proxy Pattern When:

| Scenario | Applicable Proxy Type |
|----------|-----------------------|
| Object creation is expensive and may not always be needed | Virtual |
| You need access control without modifying the subject | Protection |
| The subject lives on a remote machine or process | Remote |
| The same expensive operation is called repeatedly with the same args | Caching |
| Every operation must be audited for compliance or debugging | Logging |
| You want to add cross-cutting behavior (security, retry, metrics) without modifying business classes | Dynamic / AOP |
| The real object's interface is fixed and you cannot change it | Any |

### Do NOT Use the Proxy Pattern When:

- **The indirection cost exceeds the benefit.** For lightweight objects that are cheap to create and have no access-control requirements, a proxy adds complexity for no gain.
- **You need to change the subject's interface.** Proxy preserves the interface. If you need a different interface, use the Adapter pattern instead.
- **You are just enriching behavior for a single use case.** If there is no access-control or lifecycle concern — only behavioral enrichment — the Decorator pattern is more semantically appropriate and avoids the implication of gatekeeping.
- **You control the subject and can add the behavior directly.** Proxy shines when you cannot or should not touch the real subject.
- **Cross-cutting concerns span multiple unrelated objects.** At that scale, AOP is a cleaner solution than manually writing proxy classes per subject.

## 6. Java Implementations

### 6.1 Virtual Proxy — Lazy-Loaded Document

This example models a document management system where loading a document from the database is expensive. The `DocumentProxy` defers loading until the content is actually needed.

```java
package patterns.structural.proxy.virtual;

import java.time.Instant;
import java.util.Objects;

// ─────────────────────────────────────────────────────────────────────────────
// Subject interface — the contract shared by proxy and real object
// ─────────────────────────────────────────────────────────────────────────────

/**
 * Common abstraction for a document in the content management system.
 * Both the real implementation and the proxy implement this interface,
 * keeping the client completely unaware of lazy-loading mechanics.
 */
public interface Document {

    /**
     * Returns the full text content of the document.
     * This operation may be expensive (DB round-trip, file I/O).
     *
     * @return document content as a string
     */
    String getContent();

    /**
     * Returns lightweight metadata that is cheap to fetch.
     *
     * @return {@link DocumentMetadata} for this document
     */
    DocumentMetadata getMetadata();

    /**
     * Returns the unique identifier of this document.
     *
     * @return document ID
     */
    String getDocumentId();
}

// ─────────────────────────────────────────────────────────────────────────────
// Value object for metadata
// ─────────────────────────────────────────────────────────────────────────────

/**
 * Lightweight metadata that can be loaded quickly from a separate, indexed table.
 * Intentionally separated from content so the proxy can expose metadata
 * without triggering the expensive full-content load.
 */
public record DocumentMetadata(
        String documentId,
        String title,
        String author,
        Instant createdAt,
        long sizeInBytes
) {}

// ─────────────────────────────────────────────────────────────────────────────
// Real Subject — the expensive object
// ─────────────────────────────────────────────────────────────────────────────

/**
 * The actual document loaded from a persistent store.
 * Construction simulates a slow database round-trip.
 *
 * <p>In a real system this class would contain JPA/JDBC calls,
 * binary deserialization, decryption, etc.
 */
public class HeavyDocument implements Document {

    private final String documentId;
    private final String content;
    private final DocumentMetadata metadata;

    /**
     * Constructs a HeavyDocument by loading all data from the database.
     * This constructor simulates an expensive I/O operation (≥500 ms in production).
     *
     * @param documentId the unique identifier used to locate the document
     */
    public HeavyDocument(String documentId) {
        this.documentId = Objects.requireNonNull(documentId, "documentId must not be null");

        System.out.println("[HeavyDocument] Loading document from database: " + documentId);
        simulateDatabaseLoad();  // In production: actual JDBC/JPA call

        // Simulate loading the full content from blob storage
        this.content = "This is the full content of document [" + documentId + "]. "
                + "In production this would be megabytes of binary or text data.";

        this.metadata = new DocumentMetadata(
                documentId,
                "Annual Report 2024",
                "finance-team",
                Instant.now(),
                1_048_576L // 1 MB simulated size
        );
        System.out.println("[HeavyDocument] Document loaded successfully: " + documentId);
    }

    @Override
    public String getContent() {
        return content;
    }

    @Override
    public DocumentMetadata getMetadata() {
        return metadata;
    }

    @Override
    public String getDocumentId() {
        return documentId;
    }

    // Simulates a 200ms database round-trip
    private void simulateDatabaseLoad() {
        try {
            Thread.sleep(200);
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
            throw new RuntimeException("Document load interrupted", e);
        }
    }
}

// ─────────────────────────────────────────────────────────────────────────────
// Proxy — controls lazy initialization
// ─────────────────────────────────────────────────────────────────────────────

/**
 * Virtual proxy for {@link Document}. Defers the expensive creation of
 * {@link HeavyDocument} until the client actually requests content.
 *
 * <p><strong>Thread safety:</strong> Uses double-checked locking with a
 * {@code volatile} field to guarantee that only one thread creates the
 * real subject, even under high concurrency.
 *
 * <p>Clients that only read metadata never pay the cost of loading content.
 */
public class DocumentProxy implements Document {

    private final String documentId;

    /**
     * The real subject. Null until first access to content.
     * Declared {@code volatile} for correct double-checked locking in Java 5+.
     */
    private volatile HeavyDocument realDocument;

    /**
     * Lightweight metadata loaded eagerly from a fast index (e.g., Elasticsearch).
     * Available immediately without triggering a full document load.
     */
    private final DocumentMetadata cachedMetadata;

    /**
     * Creates a DocumentProxy. Only loads lightweight metadata eagerly.
     *
     * @param documentId the ID of the document this proxy represents
     */
    public DocumentProxy(String documentId) {
        this.documentId = Objects.requireNonNull(documentId, "documentId must not be null");
        // Metadata comes from a fast index — cheap, always available
        this.cachedMetadata = loadMetadataFromIndex(documentId);
        System.out.println("[DocumentProxy] Proxy created for: " + documentId
                + " (real document NOT loaded yet)");
    }

    /**
     * Returns the document's full content.
     * Triggers lazy initialization of the real document on first call.
     *
     * @return the full text content
     */
    @Override
    public String getContent() {
        ensureRealDocumentLoaded();
        return realDocument.getContent();
    }

    /**
     * Returns metadata. This is always served from the lightweight cache
     * and never triggers a full document load.
     *
     * @return document metadata
     */
    @Override
    public DocumentMetadata getMetadata() {
        // Intentionally does NOT call ensureRealDocumentLoaded()
        // — metadata is always available from the index
        return cachedMetadata;
    }

    @Override
    public String getDocumentId() {
        return documentId;
    }

    // ── Private helpers ──────────────────────────────────────────────────────

    /**
     * Ensures the real document is loaded exactly once.
     * Uses double-checked locking for thread safety.
     */
    private void ensureRealDocumentLoaded() {
        if (realDocument == null) {
            synchronized (this) {
                if (realDocument == null) {
                    realDocument = new HeavyDocument(documentId);
                }
            }
        }
    }

    /**
     * Simulates a fast index lookup that returns only metadata fields,
     * not the full content blob.
     */
    private DocumentMetadata loadMetadataFromIndex(String id) {
        return new DocumentMetadata(id, "Annual Report 2024",
                "finance-team", Instant.now(), 1_048_576L);
    }
}

// ─────────────────────────────────────────────────────────────────────────────
// Client / demo
// ─────────────────────────────────────────────────────────────────────────────

/**
 * Demonstrates how a client interacts with the Document interface.
 * The client is completely oblivious to lazy loading.
 */
public class VirtualProxyDemo {

    public static void main(String[] args) {
        System.out.println("=== Virtual Proxy Demo ===\n");

        // Client holds the proxy — looks exactly like a Document
        Document doc = new DocumentProxy("DOC-2024-001");

        System.out.println("\n-- Accessing metadata (fast, no DB load) --");
        DocumentMetadata meta = doc.getMetadata();
        System.out.println("Title  : " + meta.title());
        System.out.println("Author : " + meta.author());
        System.out.println("Size   : " + meta.sizeInBytes() + " bytes");

        System.out.println("\n-- Accessing content (triggers lazy load) --");
        String content = doc.getContent();
        System.out.println("Content snippet: " + content.substring(0, 60) + "...");

        System.out.println("\n-- Accessing content again (no reload, cached in real object) --");
        doc.getContent();  // fast — real document already initialized
    }
}
```

### 6.2 Protection Proxy — Role-Based Access Control

This example implements a `PayrollService` where different roles (ADMIN, HR, EMPLOYEE) have different access rights. The proxy enforces authorization before delegating.

```java
package patterns.structural.proxy.protection;

import java.math.BigDecimal;
import java.util.EnumSet;
import java.util.Map;
import java.util.Objects;
import java.util.Set;
import java.util.concurrent.ConcurrentHashMap;

// ─────────────────────────────────────────────────────────────────────────────
// Domain types
// ─────────────────────────────────────────────────────────────────────────────

/** Roles that a caller may hold in the system. */
public enum Role {
    EMPLOYEE,   // Can view only their own salary
    HR,         // Can view all salaries, update salaries
    ADMIN       // Full access including delete
}

/**
 * Immutable value object representing the security context of the current caller.
 */
public record SecurityContext(String username, Set<Role> roles) {

    public SecurityContext {
        Objects.requireNonNull(username, "username must not be null");
        roles = EnumSet.copyOf(roles); // defensive copy
    }

    /** Returns true if the caller holds at least one of the given roles. */
    public boolean hasAnyRole(Role... required) {
        for (Role r : required) {
            if (roles.contains(r)) return true;
        }
        return false;
    }
}

/** Checked exception thrown when the caller lacks sufficient permissions. */
public class AccessDeniedException extends RuntimeException {
    public AccessDeniedException(String message) {
        super(message);
    }
}

// ─────────────────────────────────────────────────────────────────────────────
// Subject interface
// ─────────────────────────────────────────────────────────────────────────────

/**
 * Contract for payroll operations.
 * Every method may throw {@link AccessDeniedException} when invoked through
 * the protection proxy with insufficient privileges.
 */
public interface PayrollService {

    /**
     * Returns the current salary for the given employee.
     *
     * @param employeeId the employee whose salary is requested
     * @return salary as a {@link BigDecimal}
     * @throws AccessDeniedException if caller lacks READ_SALARY permission
     */
    BigDecimal getSalary(String employeeId);

    /**
     * Updates the salary for the given employee.
     *
     * @param employeeId  target employee
     * @param newSalary   the new salary value; must be positive
     * @throws AccessDeniedException if caller lacks WRITE_SALARY permission
     * @throws IllegalArgumentException if newSalary is negative or zero
     */
    void updateSalary(String employeeId, BigDecimal newSalary);

    /**
     * Permanently removes an employee record.
     *
     * @param employeeId the employee to remove
     * @throws AccessDeniedException if caller lacks ADMIN permission
     */
    void deleteEmployee(String employeeId);

    /**
     * Generates and emails a payslip to the employee.
     *
     * @param employeeId target employee
     * @throws AccessDeniedException if caller lacks READ_SALARY permission
     */
    void generatePayslip(String employeeId);
}

// ─────────────────────────────────────────────────────────────────────────────
// Real Subject — pure business logic, no authorization code
// ─────────────────────────────────────────────────────────────────────────────

/**
 * Concrete payroll service implementation containing only business logic.
 * Authorization is intentionally absent — that is the proxy's responsibility.
 *
 * <p>This separation of concerns means:
 * <ul>
 *   <li>Business logic is easy to unit test without mocking security infrastructure.</li>
 *   <li>Authorization rules can change without touching payroll logic.</li>
 *   <li>The same service can be wrapped by different proxies (e.g., test proxy
 *       with no restrictions, prod proxy with full RBAC).</li>
 * </ul>
 */
public class PayrollServiceImpl implements PayrollService {

    // In-memory store simulating a database
    private final Map<String, BigDecimal> salaryStore = new ConcurrentHashMap<>(Map.of(
            "EMP-001", new BigDecimal("95000.00"),
            "EMP-002", new BigDecimal("120000.00"),
            "EMP-003", new BigDecimal("75000.00")
    ));

    @Override
    public BigDecimal getSalary(String employeeId) {
        BigDecimal salary = salaryStore.get(employeeId);
        if (salary == null) {
            throw new IllegalArgumentException("Employee not found: " + employeeId);
        }
        System.out.println("[PayrollServiceImpl] getSalary(" + employeeId + ") = " + salary);
        return salary;
    }

    @Override
    public void updateSalary(String employeeId, BigDecimal newSalary) {
        if (newSalary == null || newSalary.compareTo(BigDecimal.ZERO) <= 0) {
            throw new IllegalArgumentException("Salary must be positive");
        }
        salaryStore.put(employeeId, newSalary);
        System.out.println("[PayrollServiceImpl] updateSalary(" + employeeId
                + ", " + newSalary + ") completed");
    }

    @Override
    public void deleteEmployee(String employeeId) {
        salaryStore.remove(employeeId);
        System.out.println("[PayrollServiceImpl] deleteEmployee(" + employeeId + ") completed");
    }

    @Override
    public void generatePayslip(String employeeId) {
        BigDecimal salary = getSalary(employeeId);
        System.out.println("[PayrollServiceImpl] Generating payslip for "
                + employeeId + " — salary: " + salary);
        // In production: PDF generation, email dispatch, etc.
    }
}

// ─────────────────────────────────────────────────────────────────────────────
// Protection Proxy — enforces RBAC before delegation
// ─────────────────────────────────────────────────────────────────────────────

/**
 * Protection proxy for {@link PayrollService} that enforces role-based
 * access control before any delegation to the real service.
 *
 * <p><strong>Access matrix:</strong>
 * <pre>
 * Operation        | EMPLOYEE        | HR   | ADMIN
 * ─────────────────┼─────────────────┼──────┼──────
 * getSalary        | own record only | yes  | yes
 * updateSalary     | no              | yes  | yes
 * deleteEmployee   | no              | no   | yes
 * generatePayslip  | own record only | yes  | yes
 * </pre>
 */
public class PayrollServiceProxy implements PayrollService {

    private final PayrollService realService;
    private final SecurityContext securityContext;

    // Maps employee IDs to usernames (in production this comes from HR DB)
    private static final Map<String, String> EMPLOYEE_TO_USER = Map.of(
            "EMP-001", "alice",
            "EMP-002", "bob",
            "EMP-003", "charlie"
    );

    /**
     * Creates a PayrollServiceProxy bound to a specific security context.
     *
     * @param realService     the real payroll service to delegate to
     * @param securityContext the caller's identity and roles
     */
    public PayrollServiceProxy(PayrollService realService, SecurityContext securityContext) {
        this.realService = Objects.requireNonNull(realService, "realService must not be null");
        this.securityContext = Objects.requireNonNull(securityContext,
                "securityContext must not be null");
    }

    @Override
    public BigDecimal getSalary(String employeeId) {
        requireReadAccess(employeeId);
        return realService.getSalary(employeeId);
    }

    @Override
    public void updateSalary(String employeeId, BigDecimal newSalary) {
        requireRole(Role.HR, Role.ADMIN);
        realService.updateSalary(employeeId, newSalary);
    }

    @Override
    public void deleteEmployee(String employeeId) {
        requireRole(Role.ADMIN);
        realService.deleteEmployee(employeeId);
    }

    @Override
    public void generatePayslip(String employeeId) {
        requireReadAccess(employeeId);
        realService.generatePayslip(employeeId);
    }

    // ── Authorization helpers ────────────────────────────────────────────────

    /**
     * Enforces that the caller either holds HR/ADMIN role,
     * or is an EMPLOYEE requesting their own record.
     */
    private void requireReadAccess(String employeeId) {
        if (securityContext.hasAnyRole(Role.HR, Role.ADMIN)) {
            return; // HR and admin can read any salary
        }
        if (securityContext.hasAnyRole(Role.EMPLOYEE)) {
            String ownerUsername = EMPLOYEE_TO_USER.get(employeeId);
            if (securityContext.username().equals(ownerUsername)) {
                return; // Employee reading their own record
            }
            throw new AccessDeniedException(
                    "Employee '" + securityContext.username()
                            + "' cannot access salary for employee '" + employeeId + "'");
        }
        throw new AccessDeniedException("Access denied: insufficient role");
    }

    /**
     * Enforces that the caller holds at least one of the specified roles.
     *
     * @param required one or more roles, at least one must be held
     */
    private void requireRole(Role... required) {
        if (!securityContext.hasAnyRole(required)) {
            throw new AccessDeniedException(
                    "Access denied for user '" + securityContext.username()
                            + "'. Required role(s): " + java.util.Arrays.toString(required)
                            + ". Held roles: " + securityContext.roles());
        }
    }
}
```

### 6.3 Caching Proxy — TTL-Based API Cache

This example wraps an expensive external API call (e.g., a currency exchange rate service) with a time-to-live cache. Repeated calls within the TTL window return cached data instantly.

```java
package patterns.structural.proxy.caching;

import java.time.Duration;
import java.time.Instant;
import java.util.Map;
import java.util.Objects;
import java.util.concurrent.ConcurrentHashMap;

// ─────────────────────────────────────────────────────────────────────────────
// Domain types
// ─────────────────────────────────────────────────────────────────────────────

/**
 * Represents a currency exchange rate between two currencies.
 */
public record ExchangeRate(
        String fromCurrency,
        String toCurrency,
        double rate,
        Instant fetchedAt
) {
    @Override
    public String toString() {
        return String.format("1 %s = %.4f %s (fetched at %s)",
                fromCurrency, rate, toCurrency, fetchedAt);
    }
}

// ─────────────────────────────────────────────────────────────────────────────
// Subject interface
// ─────────────────────────────────────────────────────────────────────────────

/**
 * Contract for a service that provides foreign exchange rates.
 * Implementations may call external APIs (Bloomberg, ECB, Open Exchange Rates)
 * which have rate limits, quotas, and latency costs.
 */
public interface ExchangeRateService {

    /**
     * Returns the exchange rate between two currency codes.
     *
     * @param fromCurrency ISO 4217 currency code (e.g., "USD")
     * @param toCurrency   ISO 4217 currency code (e.g., "EUR")
     * @return the current exchange rate
     * @throws ExchangeRateException if the external API is unavailable
     */
    ExchangeRate getRate(String fromCurrency, String toCurrency);

    /**
     * Returns the current bid/ask spread for a currency pair.
     *
     * @param fromCurrency source currency
     * @param toCurrency   target currency
     * @return spread in basis points
     */
    double getSpread(String fromCurrency, String toCurrency);
}

/** Unchecked exception wrapping external API errors. */
public class ExchangeRateException extends RuntimeException {
    public ExchangeRateException(String message, Throwable cause) {
        super(message, cause);
    }
}

// ─────────────────────────────────────────────────────────────────────────────
// Real Subject — calls external API (expensive)
// ─────────────────────────────────────────────────────────────────────────────

/**
 * Real exchange rate service that calls an external REST API.
 * Each call takes ~300ms and counts against a paid API quota.
 *
 * <p>This implementation simulates the external call with a sleep and
 * a hardcoded rate table. In production this would use an HTTP client.
 */
public class OpenExchangeRateService implements ExchangeRateService {

    // Simulated rate table; in production fetched from external API
    private static final Map<String, Double> RATES = Map.of(
            "USD-EUR", 0.9234,
            "USD-GBP", 0.7891,
            "USD-JPY", 149.87,
            "EUR-GBP", 0.8546,
            "GBP-USD", 1.2673
    );

    @Override
    public ExchangeRate getRate(String fromCurrency, String toCurrency) {
        System.out.println("[OpenExchangeRateService] Calling external API for "
                + fromCurrency + " -> " + toCurrency);
        simulateApiLatency();

        String key = fromCurrency + "-" + toCurrency;
        Double rate = RATES.get(key);
        if (rate == null) {
            throw new ExchangeRateException(
                    "No rate available for pair: " + key, null);
        }
        return new ExchangeRate(fromCurrency, toCurrency, rate, Instant.now());
    }

    @Override
    public double getSpread(String fromCurrency, String toCurrency) {
        simulateApiLatency();
        return 0.0015; // 15 basis points
    }

    private void simulateApiLatency() {
        try {
            Thread.sleep(300); // Realistic REST API latency
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
            throw new ExchangeRateException("API call interrupted", e);
        }
    }
}

// ─────────────────────────────────────────────────────────────────────────────
// Cache entry — internal to the proxy
// ─────────────────────────────────────────────────────────────────────────────

/**
 * Internal cache entry that wraps a cached value with its expiry timestamp.
 *
 * @param <T> the type of the cached value
 */
record CacheEntry<T>(T value, Instant expiresAt) {

    /** Returns true if this entry is still valid at the current moment. */
    boolean isValid() {
        return Instant.now().isBefore(expiresAt);
    }
}

// ─────────────────────────────────────────────────────────────────────────────
// Caching Proxy
// ─────────────────────────────────────────────────────────────────────────────

/**
 * Caching proxy for {@link ExchangeRateService} that serves results from an
 * in-memory TTL cache, falling back to the real service only on cache misses
 * or when entries have expired.
 *
 * <p><strong>Cache key strategy:</strong> Currency pair is used as key.
 * TTL is configurable; defaults to 30 seconds (reasonable for FX rates).
 *
 * <p><strong>Thread safety:</strong> Uses {@link ConcurrentHashMap} and
 * {@code computeIfAbsent}-style semantics. Under high concurrency, multiple
 * threads might simultaneously see a cache miss and call the real service;
 * this "thundering herd" effect is acceptable here because FX rates are
 * idempotent. For scenarios requiring strict single-fetch guarantees,
 * use a per-key lock or Guava's {@code LoadingCache}.
 */
public class CachingExchangeRateServiceProxy implements ExchangeRateService {

    private final ExchangeRateService realService;
    private final Duration ttl;
    private final Map<String, CacheEntry<ExchangeRate>> rateCache  = new ConcurrentHashMap<>();
    private final Map<String, CacheEntry<Double>>        spreadCache = new ConcurrentHashMap<>();

    /** Cache statistics for observability. */
    private volatile long hits   = 0;
    private volatile long misses = 0;

    /**
     * Creates a caching proxy with the given TTL.
     *
     * @param realService the backing service to call on cache misses
     * @param ttl         how long entries remain valid before expiry
     */
    public CachingExchangeRateServiceProxy(ExchangeRateService realService, Duration ttl) {
        this.realService = Objects.requireNonNull(realService, "realService must not be null");
        this.ttl = Objects.requireNonNull(ttl, "ttl must not be null");
        System.out.printf("[CachingProxy] Initialized with TTL = %d seconds%n",
                ttl.getSeconds());
    }

    @Override
    public ExchangeRate getRate(String fromCurrency, String toCurrency) {
        String cacheKey = buildKey(fromCurrency, toCurrency);
        CacheEntry<ExchangeRate> entry = rateCache.get(cacheKey);

        if (entry != null && entry.isValid()) {
            hits++;
            System.out.printf("[CachingProxy] Cache HIT for %s (hits=%d, misses=%d)%n",
                    cacheKey, hits, misses);
            return entry.value();
        }

        // Cache miss or expired entry
        misses++;
        System.out.printf("[CachingProxy] Cache MISS for %s (hits=%d, misses=%d)%n",
                cacheKey, hits, misses);

        ExchangeRate rate = realService.getRate(fromCurrency, toCurrency);
        rateCache.put(cacheKey, new CacheEntry<>(rate, Instant.now().plus(ttl)));
        return rate;
    }

    @Override
    public double getSpread(String fromCurrency, String toCurrency) {
        String cacheKey = buildKey(fromCurrency, toCurrency);
        CacheEntry<Double> entry = spreadCache.get(cacheKey);

        if (entry != null && entry.isValid()) {
            return entry.value();
        }

        double spread = realService.getSpread(fromCurrency, toCurrency);
        spreadCache.put(cacheKey, new CacheEntry<>(spread, Instant.now().plus(ttl)));
        return spread;
    }

    /** Evicts all cache entries. Useful for testing or forced refresh. */
    public void invalidateAll() {
        rateCache.clear();
        spreadCache.clear();
        System.out.println("[CachingProxy] All cache entries invalidated");
    }

    /** Evicts the cache entry for a specific currency pair. */
    public void invalidate(String fromCurrency, String toCurrency) {
        String key = buildKey(fromCurrency, toCurrency);
        rateCache.remove(key);
        spreadCache.remove(key);
    }

    /** Returns a snapshot of current cache statistics. */
    public String getCacheStats() {
        long total = hits + misses;
        double hitRate = total == 0 ? 0 : (100.0 * hits / total);
        return String.format("Cache stats — hits: %d, misses: %d, hit rate: %.1f%%",
                hits, misses, hitRate);
    }

    private static String buildKey(String from, String to) {
        return from + "-" + to;
    }
}

// ─────────────────────────────────────────────────────────────────────────────
// Demo
// ─────────────────────────────────────────────────────────────────────────────

public class CachingProxyDemo {

    public static void main(String[] args) throws InterruptedException {
        System.out.println("=== Caching Proxy Demo ===\n");

        ExchangeRateService realService  = new OpenExchangeRateService();
        ExchangeRateService cachedService = new CachingExchangeRateServiceProxy(
                realService, Duration.ofSeconds(5));

        // First call — cache miss, calls real service (~300ms)
        long start = System.currentTimeMillis();
        ExchangeRate rate1 = cachedService.getRate("USD", "EUR");
        System.out.println(rate1 + " [took " + (System.currentTimeMillis() - start) + "ms]");

        // Second call — cache hit, instant
        start = System.currentTimeMillis();
        ExchangeRate rate2 = cachedService.getRate("USD", "EUR");
        System.out.println(rate2 + " [took " + (System.currentTimeMillis() - start) + "ms]");

        // Different pair — cache miss
        ExchangeRate rate3 = cachedService.getRate("USD", "GBP");
        System.out.println(rate3);

        // Print stats
        System.out.println("\n" + ((CachingExchangeRateServiceProxy) cachedService)
                .getCacheStats());

        // Wait for TTL to expire
        System.out.println("\nWaiting 6 seconds for TTL to expire...");
        Thread.sleep(6000);

        // Now it's a cache miss again
        ExchangeRate rate4 = cachedService.getRate("USD", "EUR");
        System.out.println(rate4 + " [TTL expired, re-fetched]");

        System.out.println("\n" + ((CachingExchangeRateServiceProxy) cachedService)
                .getCacheStats());
    }
}
```

### 6.4 Logging / Auditing Proxy

This example creates an audit trail for every call to a financial transaction service — recording the caller, timestamp, method, arguments, result, and duration.

```java
package patterns.structural.proxy.logging;

import java.math.BigDecimal;
import java.time.Instant;
import java.time.format.DateTimeFormatter;
import java.util.ArrayList;
import java.util.Collections;
import java.util.List;
import java.util.Objects;
import java.util.UUID;

// ─────────────────────────────────────────────────────────────────────────────
// Audit record
// ─────────────────────────────────────────────────────────────────────────────

/**
 * Immutable audit log entry capturing the full context of a single method call.
 * In production these are written to a tamper-evident append-only audit log
 * (e.g., Apache Kafka, AWS CloudTrail, or a dedicated audit database).
 */
public record AuditEntry(
        String traceId,
        String caller,
        Instant timestamp,
        String method,
        String arguments,
        String result,
        long durationMs,
        boolean succeeded,
        String errorMessage
) {
    @Override
    public String toString() {
        String status = succeeded ? "OK" : "FAIL";
        return String.format("[AUDIT] [%s] traceId=%s user=%s method=%s args=%s "
                        + "result=%s duration=%dms status=%s %s",
                DateTimeFormatter.ISO_INSTANT.format(timestamp),
                traceId, caller, method, arguments,
                succeeded ? result : "N/A",
                durationMs, status,
                succeeded ? "" : "error=" + errorMessage);
    }
}

// ─────────────────────────────────────────────────────────────────────────────
// Subject interface
// ─────────────────────────────────────────────────────────────────────────────

/**
 * Financial transaction service for high-value wire transfers.
 * Every operation must be audited for regulatory compliance (e.g., SOX, PCI-DSS).
 */
public interface TransactionService {

    /**
     * Initiates a wire transfer between two accounts.
     *
     * @param fromAccount source account ID
     * @param toAccount   destination account ID
     * @param amount      transfer amount; must be positive
     * @param currency    ISO 4217 currency code
     * @return the transaction ID for tracking
     */
    String initiateTransfer(String fromAccount, String toAccount,
                            BigDecimal amount, String currency);

    /**
     * Reverses a previously initiated transfer.
     *
     * @param transactionId the ID returned by {@link #initiateTransfer}
     * @return true if the reversal was successful
     */
    boolean reverseTransfer(String transactionId);

    /**
     * Returns the current balance for the given account.
     *
     * @param accountId the account to query
     * @return balance as a BigDecimal
     */
    BigDecimal getBalance(String accountId);
}

// ─────────────────────────────────────────────────────────────────────────────
// Real Subject
// ─────────────────────────────────────────────────────────────────────────────

public class TransactionServiceImpl implements TransactionService {

    @Override
    public String initiateTransfer(String fromAccount, String toAccount,
                                   BigDecimal amount, String currency) {
        // Validate amount
        if (amount == null || amount.compareTo(BigDecimal.ZERO) <= 0) {
            throw new IllegalArgumentException("Transfer amount must be positive, got: " + amount);
        }
        // In production: debit/credit database operations, compliance checks, notifications
        String txId = "TXN-" + UUID.randomUUID().toString().substring(0, 8).toUpperCase();
        System.out.println("[TransactionServiceImpl] Transfer initiated: " + txId);
        return txId;
    }

    @Override
    public boolean reverseTransfer(String transactionId) {
        if (transactionId == null || !transactionId.startsWith("TXN-")) {
            throw new IllegalArgumentException("Invalid transaction ID: " + transactionId);
        }
        System.out.println("[TransactionServiceImpl] Reversal processed for: " + transactionId);
        return true;
    }

    @Override
    public BigDecimal getBalance(String accountId) {
        // Simulate balance lookup
        return new BigDecimal("50000.00");
    }
}

// ─────────────────────────────────────────────────────────────────────────────
// Logging / Auditing Proxy
// ─────────────────────────────────────────────────────────────────────────────

/**
 * Auditing proxy for {@link TransactionService} that intercepts every method
 * call, creates an {@link AuditEntry}, delegates to the real service, and
 * appends the entry — whether the call succeeded or threw an exception.
 *
 * <p><strong>Compliance guarantee:</strong> The audit entry is written
 * <em>regardless</em> of whether the call succeeded. Failed calls produce
 * audit entries with {@code succeeded = false} and the exception message.
 *
 * <p>The {@code caller} is injected at construction time so the proxy
 * is bound to a specific authenticated user session. In practice this
 * information comes from a {@code SecurityContext} or JWT claim.
 */
public class AuditingTransactionServiceProxy implements TransactionService {

    private final TransactionService realService;
    private final String callerUsername;
    private final List<AuditEntry> auditLog = new ArrayList<>();

    /**
     * Creates an auditing proxy bound to the specified caller.
     *
     * @param realService    the backing service to delegate to
     * @param callerUsername the authenticated user making the requests
     */
    public AuditingTransactionServiceProxy(TransactionService realService,
                                            String callerUsername) {
        this.realService    = Objects.requireNonNull(realService, "realService must not be null");
        this.callerUsername = Objects.requireNonNull(callerUsername,
                "callerUsername must not be null");
    }

    @Override
    public String initiateTransfer(String fromAccount, String toAccount,
                                   BigDecimal amount, String currency) {
        String args = String.format("fromAccount=%s, toAccount=%s, amount=%s, currency=%s",
                fromAccount, toAccount, amount, currency);
        long start = System.currentTimeMillis();
        String traceId = UUID.randomUUID().toString();

        try {
            String txId = realService.initiateTransfer(fromAccount, toAccount, amount, currency);
            recordSuccess(traceId, "initiateTransfer", args, txId,
                    System.currentTimeMillis() - start);
            return txId;
        } catch (RuntimeException e) {
            recordFailure(traceId, "initiateTransfer", args, e,
                    System.currentTimeMillis() - start);
            throw e; // re-throw so caller is aware
        }
    }

    @Override
    public boolean reverseTransfer(String transactionId) {
        String args = "transactionId=" + transactionId;
        long start = System.currentTimeMillis();
        String traceId = UUID.randomUUID().toString();

        try {
            boolean result = realService.reverseTransfer(transactionId);
            recordSuccess(traceId, "reverseTransfer", args, String.valueOf(result),
                    System.currentTimeMillis() - start);
            return result;
        } catch (RuntimeException e) {
            recordFailure(traceId, "reverseTransfer", args, e,
                    System.currentTimeMillis() - start);
            throw e;
        }
    }

    @Override
    public BigDecimal getBalance(String accountId) {
        String args = "accountId=" + accountId;
        long start = System.currentTimeMillis();
        String traceId = UUID.randomUUID().toString();

        try {
            BigDecimal balance = realService.getBalance(accountId);
            recordSuccess(traceId, "getBalance", args, balance.toString(),
                    System.currentTimeMillis() - start);
            return balance;
        } catch (RuntimeException e) {
            recordFailure(traceId, "getBalance", args, e,
                    System.currentTimeMillis() - start);
            throw e;
        }
    }

    // ── Audit helpers ────────────────────────────────────────────────────────

    private void recordSuccess(String traceId, String method, String args,
                                String result, long durationMs) {
        AuditEntry entry = new AuditEntry(
                traceId, callerUsername, Instant.now(),
                method, args, result, durationMs, true, null);
        auditLog.add(entry);
        System.out.println(entry);
    }

    private void recordFailure(String traceId, String method, String args,
                                RuntimeException e, long durationMs) {
        AuditEntry entry = new AuditEntry(
                traceId, callerUsername, Instant.now(),
                method, args, null, durationMs, false, e.getMessage());
        auditLog.add(entry);
        System.err.println(entry);
    }

    /**
     * Returns an unmodifiable view of all audit entries recorded by this proxy.
     * In production, entries would be persisted, not kept in memory.
     *
     * @return immutable list of audit entries
     */
    public List<AuditEntry> getAuditLog() {
        return Collections.unmodifiableList(auditLog);
    }
}

// ─────────────────────────────────────────────────────────────────────────────

public class AuditingProxyDemo {

        System.out.println("=== Logging / Auditing Proxy Demo ===\n");

            System.out.println("Expected error caught by client: " + e.getMessage());

        System.out.println("\n=== Full Audit Trail ===");

    }
}
```

### 6.5 Dynamic Proxy — java.lang.reflect.Proxy

Java's built-in dynamic proxy creates proxy objects at runtime without writing a concrete proxy class. This is the mechanism underlying Spring AOP, Hibernate lazy loading, and many DI frameworks.

```java
package patterns.structural.proxy.dynamic;

import java.lang.reflect.InvocationHandler;
import java.lang.reflect.Method;
import java.lang.reflect.Proxy;
import java.time.Instant;
import java.util.Arrays;

// ─────────────────────────────────────────────────────────────────────────────
// Subject interface
// ─────────────────────────────────────────────────────────────────────────────

/**
 * A service interface that we want to proxy dynamically.
 * Note: java.lang.reflect.Proxy requires the subject to be an interface.
 */
public interface UserService {
    String findById(String userId);
    void createUser(String userId, String email);
    void deleteUser(String userId);
}

// ─────────────────────────────────────────────────────────────────────────────
// Real Subject
// ─────────────────────────────────────────────────────────────────────────────

public class UserServiceImpl implements UserService {

    @Override
    public String findById(String userId) {
        return "{\"id\":\"" + userId + "\",\"email\":\"user@example.com\"}";
    }

    @Override
    public void createUser(String userId, String email) {
        System.out.println("[UserServiceImpl] Created user: " + userId + " / " + email);
    }

    @Override
    public void deleteUser(String userId) {
        System.out.println("[UserServiceImpl] Deleted user: " + userId);
    }
}

// ─────────────────────────────────────────────────────────────────────────────
// Generic Logging InvocationHandler
// ─────────────────────────────────────────────────────────────────────────────

/**
 * A reusable {@link InvocationHandler} that logs every method call with timing
 * and then delegates to the wrapped target object.
 *
 * <p>This single handler class can proxy ANY interface without writing
 * interface-specific proxy code. This is the foundation of Spring AOP's
 * JDK proxy strategy.
 *
 * <p><strong>How it works:</strong>
 * <ol>
 *   <li>Client calls a method on the proxy object.</li>
 *   <li>JVM routes the call to {@link #invoke(Object, Method, Object[])}.</li>
 *   <li>The handler performs pre-processing (logging, auth, caching).</li>
 *   <li>The handler calls {@code method.invoke(target, args)} to delegate.</li>
 *   <li>The handler performs post-processing with the result.</li>
 * </ol>
 */
public class LoggingInvocationHandler implements InvocationHandler {

    private final Object target;
    private final String proxyName;

    /**
     * Constructs a logging handler for the given target object.
     *
     * @param target    the real object to delegate to
     * @param proxyName a human-readable label for log messages
     */
    public LoggingInvocationHandler(Object target, String proxyName) {
        this.target    = target;
        this.proxyName = proxyName;
    }

    /**
     * Intercepts every method call on the proxy.
     *
     * @param proxy  the proxy instance (rarely used)
     * @param method the method being called
     * @param args   the arguments to the method (null for zero-arg methods)
     * @return the result from the real target
     * @throws Throwable any exception thrown by the real target
     */
    @Override
    public Object invoke(Object proxy, Method method, Object[] args) throws Throwable {
        String methodName = proxyName + "." + method.getName();
        String argsStr    = args == null ? "()" : Arrays.toString(args);
        Instant start     = Instant.now();

        System.out.printf("[DynamicProxy] --> %s%s at %s%n", methodName, argsStr, start);

        try {
            Object result = method.invoke(target, args);
            long durationMs = java.time.Duration.between(start, Instant.now()).toMillis();

            System.out.printf("[DynamicProxy] <-- %s returned: %s [%dms]%n",
                    methodName, result, durationMs);
            return result;

        } catch (java.lang.reflect.InvocationTargetException e) {
            // Unwrap the real exception so callers receive it correctly
            Throwable cause = e.getCause();
            long durationMs = java.time.Duration.between(start, Instant.now()).toMillis();
            System.err.printf("[DynamicProxy] <-- %s threw %s: %s [%dms]%n",
                    methodName, cause.getClass().getSimpleName(),
                    cause.getMessage(), durationMs);
            throw cause; // re-throw unwrapped
        }
    }
}

// ─────────────────────────────────────────────────────────────────────────────
// Generic Proxy Factory
// ─────────────────────────────────────────────────────────────────────────────

/**
 * Factory that creates JDK dynamic proxies with a logging handler.
 * Demonstrates the typical "proxy factory" pattern used in frameworks.
 *
 * <p>Spring's {@code ProxyFactory} and {@code AopProxyFactory} follow
 * the same structural idea, but with a much richer advice chain.
 */
public final class LoggingProxyFactory {

    private LoggingProxyFactory() { /* utility class */ }

    /**
     * Wraps the given target in a JDK dynamic proxy that logs all method calls.
     *
     * @param target          the real service to wrap
     * @param serviceInterface the interface the proxy will implement
     * @param <T>             the service type
     * @return a proxy implementing {@code serviceInterface}
     */
    @SuppressWarnings("unchecked")
    public static <T> T createLoggingProxy(T target, Class<T> serviceInterface) {
        return (T) Proxy.newProxyInstance(
                serviceInterface.getClassLoader(),
                new Class<?>[]{ serviceInterface },
                new LoggingInvocationHandler(target, serviceInterface.getSimpleName())
        );
    }
}

// ─────────────────────────────────────────────────────────────────────────────
// Chained handlers — multiple cross-cutting concerns composed
// ─────────────────────────────────────────────────────────────────────────────

/**
 * Demonstrates how multiple InvocationHandlers can be chained (Proxy wrapping Proxy)
 * to compose independent cross-cutting concerns — the same principle used by
 * Spring AOP's interceptor chain.
 *
 * <p>The chain: Client → AuthHandler → LoggingHandler → RealService
 */
public class AuthorizationInvocationHandler implements InvocationHandler {

    private final Object target;
    private final String currentUser;
    private final java.util.Set<String> protectedMethods;

    public AuthorizationInvocationHandler(Object target, String currentUser,
                                           String... protectedMethods) {
        this.target           = target;
        this.currentUser      = currentUser;
        this.protectedMethods = new java.util.HashSet<>(Arrays.asList(protectedMethods));
    }

    @Override
    public Object invoke(Object proxy, Method method, Object[] args) throws Throwable {
        if (protectedMethods.contains(method.getName())) {
            System.out.printf("[AuthHandler] Checking authorization for %s() by user '%s'%n",
                    method.getName(), currentUser);
            if (!"admin".equals(currentUser)) {
                throw new SecurityException(
                        "User '" + currentUser + "' is not authorized to call "
                                + method.getName());
            }
        }
        return method.invoke(target, args);
    }
}

// ─────────────────────────────────────────────────────────────────────────────
// Demo
// ─────────────────────────────────────────────────────────────────────────────

public class DynamicProxyDemo {

    public static void main(String[] args) {
        System.out.println("=== Dynamic Proxy Demo ===\n");

        UserService realService = new UserServiceImpl();

        // ── Single logging proxy ──────────────────────────────────────────────
        System.out.println("--- Single Logging Proxy ---");
        UserService loggingProxy = LoggingProxyFactory.createLoggingProxy(
                realService, UserService.class);

        loggingProxy.findById("USR-123");
        loggingProxy.createUser("USR-456", "new@example.com");

        // ── Chained proxy: Auth wrapping LoggingProxy wrapping Real ───────────
        System.out.println("\n--- Chained Proxy (Auth + Logging) as non-admin user ---");

        // Layer 1: Logging proxy around real service
        UserService withLogging = LoggingProxyFactory.createLoggingProxy(
                realService, UserService.class);

        // Layer 2: Auth proxy around the logging proxy
        UserService withAuthAndLogging = (UserService) Proxy.newProxyInstance(
                UserService.class.getClassLoader(),
                new Class<?>[]{ UserService.class },
                new AuthorizationInvocationHandler(
                        withLogging, "regular-user", "deleteUser", "createUser")
        );

        withAuthAndLogging.findById("USR-789");  // Allowed

        try {
            withAuthAndLogging.deleteUser("USR-001");  // Blocked by auth
        } catch (SecurityException e) {
            System.out.println("Caught: " + e.getMessage());
        }

        // ── Admin user — full access ──────────────────────────────────────────
        System.out.println("\n--- Chained Proxy as admin user ---");
        UserService adminService = (UserService) Proxy.newProxyInstance(
                UserService.class.getClassLoader(),
                new Class<?>[]{ UserService.class },
                new AuthorizationInvocationHandler(
                        withLogging, "admin", "deleteUser", "createUser")
        );

        adminService.deleteUser("USR-001");  // Allowed
    }
}
```

## 7. Real-World Examples in the Java Ecosystem

### 7.1 java.lang.reflect.Proxy

Java's built-in dynamic proxy mechanism (introduced in Java 1.3) is the canonical example. It can proxy any interface at runtime:

```java
// Every Spring @Service bean annotated with @Transactional is actually
// wrapped in a proxy like this at application startup:

Object proxy = Proxy.newProxyInstance(
    beanClass.getClassLoader(),
    beanClass.getInterfaces(),          // e.g., [UserService.class]
    new TransactionInterceptor(...)     // the "advice" in AOP terms
);
```

The proxy intercepts each method call, checks if it requires transaction management (based on `@Transactional` metadata), begins/commits/rolls back the transaction, and then delegates.

**Key constraint:** JDK proxies only work for interfaces. For concrete classes, Spring falls back to CGLIB (subclass-based proxy).

---

### 7.2 Spring AOP — @Transactional, @Cacheable, @Secured

Spring's entire AOP infrastructure is built on the Proxy pattern:

| Annotation | Proxy behavior |
|------------|----------------|
| `@Transactional` | Proxy starts a transaction before the method, commits on success, rolls back on exception |
| `@Cacheable("prices")` | Proxy checks the cache before the method; if hit, returns cached value without calling the real method |
| `@CacheEvict` | Proxy calls the real method, then evicts specified cache entries |
| `@Secured("ROLE_ADMIN")` | Proxy checks `SecurityContextHolder` for the required role; throws `AccessDeniedException` if absent |
| `@Async` | Proxy intercepts the call and submits it to a `TaskExecutor` instead of running synchronously |
| `@Retryable` | Proxy wraps the call in a retry loop with configurable backoff |

---

### 7.3 Hibernate Lazy Loading (Virtual Proxy)

Hibernate implements the Virtual Proxy pattern for all lazily-fetched associations:

```java
@Entity
public class Order {

    @ManyToOne(fetch = FetchType.LAZY)  // Hibernate creates a proxy for Customer
    private Customer customer;          // This field holds a proxy, not a real Customer

    // When your code calls order.getCustomer().getName(),
    // Hibernate's proxy intercepts .getName(), fires a SELECT, populates the real
    // object, and returns the result. All subsequent calls skip the SELECT.
}
```

The proxy Hibernate generates is a CGLIB subclass of `Customer`. It overrides every method to check whether the entity has been initialized. The `customer` field in `Order` points to this CGLIB subclass instance, not a real `Customer`. This is a textbook Virtual Proxy — the real object (populated entity) is created on first access.

**The famous LazyInitializationException** is what happens when a Hibernate proxy is accessed after the `Session` is closed. The proxy exists, but it cannot fire the SQL because there is no open session. This is a direct consequence of the Virtual Proxy pattern's dependency on the factory (Session) that can create the real object.

### 7.4 gRPC Client Stubs (Remote Proxy)

When you run `protoc` to generate gRPC code, the generated stub is a Remote Proxy:

```java
// Generated stub — this IS the remote proxy
ManagedChannel channel = ManagedChannelBuilder
        .forAddress("payments-service", 50051)
        .usePlaintext()
        .build();

PaymentsGrpc.PaymentsBlockingStub stub = PaymentsGrpc.newBlockingStub(channel);

// From the caller's perspective this looks like a local call:
PaymentResponse response = stub.processPayment(PaymentRequest.newBuilder()
        .setAmount(100_00L)
        .setCurrency("USD")
        .build());
```

The stub handles protobuf serialization, connection management, retries, and deadline propagation — all invisible to the caller.

### 7.5 Spring Data JPA Repository Proxies

When you declare a Spring Data interface:

```java
public interface UserRepository extends JpaRepository<User, Long> {
    List<User> findByEmail(String email);
}
```

Spring Data creates a JDK dynamic proxy at startup that implements the `UserRepository` interface. The `InvocationHandler` (internally `SimpleJpaRepository` delegation logic) inspects the method name, parses the query DSL, generates JPQL, and executes it. You never write a class — only an interface. The implementation IS a proxy.

## 8. Spring AOP as a Proxy Application

### How @Transactional Actually Works Under the Hood

Understanding `@Transactional` as a proxy is one of the most practical applications of the Proxy pattern in enterprise Java.

#### Step 1 — Bean creation and proxy wrapping

When Spring's ApplicationContext initializes, `BeanPostProcessor` implementations inspect each bean. `InfrastructureAdvisorAutoProxyCreator` (or `AnnotationAwareAspectJAutoProxyCreator`) detects that `UserService` has `@Transactional` methods and wraps it in a proxy:

```
Original bean:
  UserServiceImpl (your code)

After post-processing:
  Proxy (JDK proxy OR CGLIB subclass)
      └── TransactionInterceptor (the "around" advice)
              └── UserServiceImpl (the real object)
```

#### Step 2 — The interceptor chain on each call

```
Client code:
  userService.createUser(dto)  ← userService is the PROXY, not the real object

Proxy intercepts and runs TransactionInterceptor.invoke():
  1. Inspect @Transactional metadata on method
  2. Resolve transaction attributes (propagation, isolation, readOnly, rollbackFor)
  3. Call TransactionManager.getTransaction(attributes)  ← BEGIN TRANSACTION
  4. Invoke the real method:  userServiceImpl.createUser(dto)
     a. If no exception → TransactionManager.commit()
     b. If RuntimeException (or matching rollbackFor) → TransactionManager.rollback()
     c. If checked exception NOT in rollbackFor → commit anyway (surprising gotcha!)
  5. Return the result to the caller
```

#### Step 3 — The self-invocation gotcha

This is the most important pitfall from a proxy standpoint:

```java
@Service
public class OrderService {

    @Transactional
    public void placeOrder(Order order) {
        // ... business logic ...
        this.sendConfirmationEmail(order);  // BUG: this bypasses the proxy!
    }

    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void sendConfirmationEmail(Order order) {
        // This expects a NEW transaction, but it runs in the SAME transaction
        // as placeOrder() because 'this' is the real object, not the proxy.
    }
}
```

When `placeOrder` calls `this.sendConfirmationEmail(...)`, it is calling the real `OrderService` object directly, bypassing the proxy. The `@Transactional` on `sendConfirmationEmail` has no effect for self-invocations.

**Fix:** Inject the bean as a dependency (getting the proxy), or use `AopContext.currentProxy()`:

```java
@Transactional
public void placeOrder(Order order) {
    // ...
    // Get the proxy so @Transactional is honoured:
    ((OrderService) AopContext.currentProxy()).sendConfirmationEmail(order);
}
```

#### CGLIB vs JDK Proxy selection

Spring chooses the proxy strategy based on the bean's type:

| Condition | Strategy |
|-----------|----------|
| Bean implements at least one interface | JDK dynamic proxy (default) |
| Bean is a concrete class with no interface | CGLIB subclass proxy |
| `@EnableTransactionManagement(proxyTargetClass = true)` | Always CGLIB |
| Spring Boot default (2.x+) | Always CGLIB |

CGLIB proxies create a subclass of your class and override every non-final method. This is why:
- `final` methods cannot be proxied by CGLIB (no override possible).
- `final` classes cannot be proxied by CGLIB at all.
- `private` methods cannot be intercepted regardless of proxy type.

## 9. Proxy vs Decorator — Detailed Comparison

### Structural Similarity

At the code level, Proxy and Decorator are nearly identical:

```
Both patterns:
  - Implement the same interface as the wrapped object
  - Hold a reference to the wrapped object
  - Delegate method calls to the wrapped object
  - May add behavior before/after delegation
```

A Proxy class and a Decorator class look so alike that if you saw only the Java source code (without context), it would be difficult to determine which pattern it implements.

### Intent Difference — The Fundamental Distinction

| Dimension | Proxy | Decorator |
|-----------|-------|-----------|
| **Primary goal** | Control ACCESS to the subject | Enhance BEHAVIOR of the subject |
| **Relationship to subject** | Manages the subject's lifecycle or guards its gate | Assumes subject already exists and is accessible |
| **Who creates the real object** | Often the proxy itself (Virtual Proxy) | Client creates both; Decorator just wraps |
| **Client awareness** | Client usually does NOT know it has a proxy | Client often chooses which decorators to stack |
| **Stacking / composition** | Rarely stacked; one proxy per subject is typical | Designed to be stacked arbitrarily (BufferedInputStream → FileInputStream) |
| **Focus** | WHO can access the object, WHEN it is created, WHERE it lives | WHAT the object does when called |
| **GoF description** | "Control access to an object" | "Attach additional responsibilities to an object dynamically" |

### Detailed Example Contrasting Intent

```java
// PROXY — controls whether you get the report at all
public class ReportProxy implements ReportService {
    private final ReportService real;
    private final SecurityContext ctx;

    @Override
    public Report generate(String reportId) {
        if (!ctx.hasRole(Role.ANALYST)) {
            throw new AccessDeniedException("Reports require ANALYST role");
        }
        return real.generate(reportId);  // gatekeeping
    }
}

// DECORATOR — adds formatting on top of any ReportService
public class PdfFormattingReportDecorator implements ReportService {
    private final ReportService wrapped;  // could be real or another decorator

    @Override
    public Report generate(String reportId) {
        Report raw = wrapped.generate(reportId);  // get the base report
        return applyPdfFormatting(raw);            // enhance the behavior
    }
}
```

The Proxy asks: **"Should this call be allowed to proceed, and in what form?"**

The Decorator asks: **"How do we enrich the result or execution of this call?"**

### When They Blur Together

In practice the lines blur in two situations:

1. **Logging/Auditing Proxy** — The audit proxy does not control access; it enriches the process by recording it. This is structurally and intentionally closer to a Decorator. Many practitioners correctly call an audit wrapper a Decorator. The naming choice rarely matters in code review, but it matters in communication.

2. **Spring AOP** — Spring's `@Transactional` proxy controls a resource (the database transaction) around the call, which is more Proxy-like. Spring's `@Cacheable` skips the real call on a cache hit, which feels like access control. Yet both are implemented as "around advice" which is the Decorator structure.

**The rule of thumb:** If the wrapper's primary concern is whether and how the subject is reached (lifecycle, security, location, creation), call it a Proxy. If the primary concern is what additional processing happens around the call (formatting, logging, retry), call it a Decorator.

## 10. Trade-offs

### Pros

| Benefit | Explanation |
|---------|-------------|
| **Separation of concerns** | Cross-cutting concerns (auth, caching, logging) live outside business classes |
| **Single Responsibility** | The real subject focuses solely on its business logic |
| **Open/Closed Principle** | You can add new behaviors (new proxy type) without modifying existing classes |
| **Lazy initialization** | Expensive objects are not created until actually needed, saving startup time and memory |
| **Transparent to clients** | Clients coded against the Subject interface work with proxies without any change |
| **Testability** | Business logic can be unit-tested without security or caching infrastructure |
| **Composability** | Dynamic proxies and interceptor chains allow composing multiple cross-cutting concerns |

### Cons

| Cost | Explanation |
|------|-------------|
| **Indirection overhead** | Every method call goes through an extra layer; in hot paths this adds measurable latency |
| **Complexity for readers** | A developer reading a call stack sees the proxy frames and must understand the proxy-real distinction |
| **Dynamic proxy limitations** | JDK proxies require interfaces; CGLIB proxies cannot intercept final/private methods |
| **Self-invocation bypass** | Internal `this.method()` calls skip the proxy entirely — a frequent source of `@Transactional` bugs |
| **Debugging difficulty** | Stack traces include generated proxy classes (e.g., `$Proxy47`, `UserService$$EnhancerBySpringCGLIB`) making them harder to read |
| **Proliferation of classes** | For N interfaces, you may need N proxy classes (mitigated by dynamic proxies, but still conceptual overhead) |
| **Proxy leakage** | If the real subject escapes (returned by a method that returns `this`, for example), the proxy is bypassed |
| **Equals/hashCode issues** | If `equals()` is not overridden to account for the proxy wrapper, identity comparisons can fail unexpectedly |

## 11. Common Pitfalls and Gotchas

### Pitfall 1 — Self-Invocation in Spring AOP

The most common Spring bug caused by Proxy mechanics:

```java
@Service
public class InventoryService {

    @Cacheable("inventory")
    public int getStock(String productId) { /* DB query */ return 42; }

    public void processOrder(String productId) {
        // WRONG: calls this.getStock(), bypasses the caching proxy
        int stock = this.getStock(productId);

        // RIGHT: inject self via @Autowired private InventoryService self;
        // int stock = self.getStock(productId);
    }
}
```

### Pitfall 2 — Final Methods and Classes with CGLIB

```java
@Service
public class PaymentService {

    @Transactional
    public final void processPayment() {
        // WARNING: @Transactional has NO effect here when using CGLIB proxy.
        // CGLIB cannot override final methods.
    }
}
```

Spring will not warn you about this at startup. The transaction annotation is silently ignored.

### Pitfall 3 — Casting the Proxy

```java
UserService service = applicationContext.getBean(UserService.class);
// service is a proxy, not a UserServiceImpl

// This will throw ClassCastException:
UserServiceImpl impl = (UserServiceImpl) service; // WRONG

// Correct: cast to the interface
UserService svc = (UserService) service;          // OK

// Or use AopUtils to check:
boolean isProxy = AopUtils.isAopProxy(service);
Class<?> targetClass = AopUtils.getTargetClass(service);
```

### Pitfall 4 — equals() and hashCode() Through Proxy

If you place a Spring bean (a proxy) into a `HashSet` or use it as a Map key, and the `equals()`/`hashCode()` contract depends on the concrete type, you may get surprising behavior because the proxy overrides identity without overriding value equality.

### Pitfall 5 — Thread Safety in Virtual Proxy

A naive Virtual Proxy without proper synchronization can create the real object multiple times under concurrent access:

```java
// WRONG — race condition: two threads may both see null and create HeavyObject
public SomeData getData() {
    if (realObject == null) {
        realObject = new HeavyObject();  // created twice in race
    }
    return realObject.getData();
}

// CORRECT — double-checked locking with volatile
private volatile HeavyObject realObject;

public SomeData getData() {
    if (realObject == null) {
        synchronized (this) {
            if (realObject == null) {
                realObject = new HeavyObject();
            }
        }
    }
    return realObject.getData();
}
```

### Pitfall 6 — Checked Exceptions in Dynamic Proxies

When using `java.lang.reflect.Proxy`, the `InvocationHandler.invoke()` declares `throws Throwable`. If you re-throw the `InvocationTargetException` without unwrapping, callers receive a wrapped exception instead of the original:

```java
// WRONG — caller receives InvocationTargetException, not the real exception
throw e;

// CORRECT — unwrap and re-throw the actual cause
throw e.getCause();
```

## 13. Interview Questions with Model Answers

---

### Q1. What is the Proxy design pattern and what problem does it solve?

**Model Answer:**

The Proxy pattern introduces a surrogate object that stands in for a real object and controls access to it. Both the proxy and the real object implement the same interface, so clients interact with the proxy identically to how they would interact with the real object.

The problem it solves is the need to add logic around access to an object without modifying the object itself. This access logic might include:

- Deferring expensive object creation until the object is actually needed (Virtual Proxy).
- Enforcing authorization rules before allowing calls (Protection Proxy).
- Transparently routing calls to a remote process (Remote Proxy).
- Returning cached results for repeated calls (Caching Proxy).
- Recording every call for compliance or debugging (Logging Proxy).

Without the Proxy pattern, this logic would need to live either inside the real object (violating Single Responsibility) or at every call site in the client code (violating DRY and Open/Closed). The proxy centralizes this concern in one place.

---

### Q2. Explain how Spring's @Transactional annotation works under the hood using the Proxy pattern.

**Model Answer:**

When a Spring application starts, `BeanPostProcessor` implementations scan every bean for AOP-applicable annotations including `@Transactional`. When a bean has transactional methods, Spring wraps it in a proxy before returning it from the application context.

The proxy is either a JDK dynamic proxy (if the bean implements at least one interface) or a CGLIB subclass (for concrete classes, and as the Spring Boot default). The proxy is the object that gets injected wherever your `@Autowired` dependencies point.

When a client calls a method on the injected bean, the call hits the proxy first. The proxy's `TransactionInterceptor` (an `MethodInterceptor` in Spring AOP terms) executes an around advice:

1. It inspects the `@Transactional` metadata on the method — propagation, isolation level, `readOnly` flag, `rollbackFor`, `noRollbackFor`.
2. It asks the `PlatformTransactionManager` (JpaTransactionManager, DataSourceTransactionManager, etc.) to begin or join a transaction according to the propagation rules.
3. It invokes the real method via the interceptor chain.
4. If the method returns normally, it commits the transaction.
5. If a `RuntimeException` (or any exception listed in `rollbackFor`) is thrown, it rolls back.
6. Critically, if a checked exception NOT in `rollbackFor` is thrown, it still commits — a common surprise for developers.

The most important gotcha: if a method inside the same class calls another `@Transactional` method via `this.method()`, the call bypasses the proxy and the transaction annotation on the called method is completely ignored. This is because `this` refers to the real object, not the proxy. The fix is to inject the bean reference (getting the proxy) or to use `AopContext.currentProxy()`.

### Q3. What is the difference between Virtual Proxy and Lazy Initialization? Are they the same thing?

**Model Answer:**

They are related concepts but not the same thing. Lazy initialization is a general technique of deferring object creation or computation until it is first needed. Virtual Proxy is a design pattern that implements lazy initialization behind an interface, making it transparent to the client.

With plain lazy initialization, the client code is responsible for checking whether the object exists and creating it if not. This mixes lifecycle management into business code and is not transparent — every client must participate.

```java
// Plain lazy initialization — client is responsible
if (heavyObject == null) {
    heavyObject = new HeavyObject();
}
heavyObject.doWork();
```

With a Virtual Proxy, the proxy object handles the lazy initialization check internally. The client holds a reference to the Subject interface and calls methods on it without any awareness that the real object may not yet exist:

```java
// Virtual Proxy — client is unaware
Document doc = new DocumentProxy("DOC-001");  // no DB load yet
doc.getMetadata();    // fast, served from index
doc.getContent();     // triggers lazy load transparently
```

The distinction matters because the Virtual Proxy also enforces the object identity contract — the client always holds the same proxy reference, and the proxy creates the real object at most once (if designed correctly, using double-checked locking for thread safety). Hibernate's lazy-loaded entities are the canonical production example: `order.getCustomer()` is a proxy, and calling `.getName()` on it triggers the database load transparently.

---

### Q4. Why can't Spring's @Transactional proxy intercept private or final methods? What are the implications?

**Model Answer:**

Spring uses two proxy strategies: JDK dynamic proxies and CGLIB subclass proxies.

JDK dynamic proxies generate a class that implements the target interface and dispatches all interface method calls to the `InvocationHandler`. Since private methods are not part of any interface, they cannot appear in the proxy's interface implementation and therefore cannot be intercepted.

CGLIB proxies generate a subclass of the target class and override each non-final, non-private method. The key word is "override" — Java's method override mechanism only works for non-final methods. A `final` method cannot be overridden, so CGLIB cannot insert advice around it. Similarly, private methods have no override mechanics.

The implication is that `@Transactional`, `@Cacheable`, `@Async`, and any other annotation that Spring implements via AOP simply has no effect on private or final methods. Spring will not emit a compile-time error or even a runtime warning for this misconfiguration. The annotation is silently ignored.

Practical rule: never put `@Transactional` (or any Spring AOP annotation) on private methods. For transaction demarcation of private logic, either make the method package-private or protected, or extract it into a separate Spring bean.

For truly internal implementation details that should not be proxied (e.g., helper methods), private is correct — just do not annotate them with AOP annotations.

### Q5. When would you choose a Dynamic Proxy (java.lang.reflect.Proxy) over writing a concrete static proxy class?

**Model Answer:**

The choice between dynamic and static (concrete) proxies involves several trade-offs.

**Choose a dynamic proxy when:**

The number of interfaces to proxy is large or unknown at compile time. Writing a concrete proxy class for each of twenty service interfaces would produce a maintenance burden: every time an interface changes, the proxy must also change. A single generic `InvocationHandler` handles all of them.

The cross-cutting concern is the same for all methods, regardless of which interface they belong to. Logging, metrics collection, and retry logic are uniform across methods — a single handler implementation serves all.

The framework architecture requires working with arbitrary interfaces at runtime. Spring AOP, gRPC client stubs, Spring Data repositories, and Mockito's mocking engine all use dynamic proxies because they cannot know at compile time which interfaces they will proxy.

**Choose a static (concrete) proxy class when:**

Only a few interfaces need proxying and the interface is stable. Static proxies are type-safe, refactor safely with IDE tooling, and produce clear stack traces with no reflection overhead.

The per-method logic differs significantly. If `getSalary()`, `updateSalary()`, and `deleteEmployee()` require different authorization rules, encoding them in a generic `invoke()` method becomes complex and fragile. A concrete proxy class with explicit method bodies is more readable and maintainable.

You want compile-time verification that the proxy class implements the full interface. A static proxy will fail to compile if a new method is added to the interface without being added to the proxy. A dynamic proxy silently forwards the call without any check at compile time.

Performance is critical. Reflection in `Method.invoke()` adds overhead (typically 10–50ns per call). In hot paths (tens of millions of calls per second), this overhead is measurable. Static proxies incur only normal virtual dispatch overhead.

### Q6. How would you design a thread-safe Virtual Proxy for an expensive resource? What pattern ensures the real object is created at most once?

**Model Answer:**

The standard approach is double-checked locking (DCL) combined with a `volatile` field. This pattern has been correct and safe in Java since Java 5's revised memory model (JSR-133), which gives `volatile` strong visibility guarantees.

```java
public class ResourceProxy implements Resource {
    private volatile Resource realResource;  // volatile is critical

    @Override
    public void doWork() {
        if (realResource == null) {                  // First check — no lock
            synchronized (this) {
                if (realResource == null) {          // Second check — under lock
                    realResource = new HeavyResource();
                }
            }
        }
        realResource.doWork();
    }
}
```

The first `null` check avoids acquiring the lock for every call once initialized. The `synchronized` block ensures only one thread creates the object. The second `null` check inside the synchronized block handles the case where multiple threads passed the first check simultaneously — only the first thread to acquire the lock creates the object; subsequent threads find it already set.

The `volatile` keyword is not optional. Without it, the Java Memory Model permits the JIT compiler and CPU to reorder the write to `realResource` relative to the write to its internal fields. A second thread could see a non-null but partially constructed object. `volatile` establishes a happens-before relationship that ensures the fully constructed object is visible to all threads after the write.

An alternative for non-subclassable scenarios is the initialization-on-demand holder idiom (lazy class loading by the JVM):

```java
public class ResourceProxy implements Resource {
    private static class Holder {
        static final Resource REAL = new HeavyResource();
    }
    @Override
    public void doWork() {
        Holder.REAL.doWork();
    }
}
```

The JVM guarantees that class initialization is thread-safe, so `HeavyResource` is created exactly once when `Holder` is first loaded.

### Q7. Explain the difference between Proxy and Decorator patterns. Give an example where the distinction is meaningful.

**Model Answer:**

Proxy and Decorator are the most frequently confused GoF patterns because they share identical structure: both implement the same interface as the wrapped object, both hold a reference to the wrapped object, and both delegate calls to it.

The distinction is one of intent and relationship to the wrapped object:

A **Proxy** controls access to its subject. It cares about whether and how the subject is reached. The proxy may decide not to forward a call (protection proxy denies access), may delay creation of the subject (virtual proxy defers instantiation), or may route calls to a remote subject. Importantly, the proxy often manages the subject's lifecycle — it may be the one that creates the real object.

A **Decorator** enriches its subject's behavior. It always forwards the call but may do additional work before or after delegation. It never blocks access; it assumes the subject exists, is accessible, and should be called. Decorators are typically designed to be composable: `BufferedReader(new InputStreamReader(new FileInputStream(...)))` stacks three decorators where each adds one capability.

A concrete example where the distinction matters:

**Logging Proxy vs Logging Decorator:** An audit log that records "this user attempted to call deleteEmployee but was blocked" is a Proxy (access control is happening). An audit log that records "deleteEmployee was called with args X and returned Y" is more of a Decorator (it enriches the execution, all calls succeed in reaching the real method).

**Practical implication:** When designing a caching wrapper, ask: "Does a cache hit mean the real method should not be called?" If yes, it is Proxy behavior (access is controlled — the client is given cached data instead of being allowed through to the real subject). This explains why `@Cacheable` in Spring is technically implemented as a proxy even though it looks like a decorator.

For senior engineers, the naming is less important than the reasoning: can you explain why you chose one over the other for a given use case?

### Q8. You are reviewing code where a developer has annotated a method with @Transactional but the transactions are not working correctly. Walk me through your debugging process.

**Model Answer:**

I follow a systematic checklist based on the known failure modes of Spring's Proxy-based AOP.

**Step 1 — Is the bean actually being proxied?**
I would confirm the bean is managed by Spring. If it was instantiated with `new` rather than injected, there is no proxy and no transaction management. I add a log statement or use the debugger to check the class name of the injected bean — a proxied bean's class will contain `$$EnhancerBySpringCGLIB` or `$Proxy`.

**Step 2 — Self-invocation?**
I check whether the transactional method is called from within the same class. If `methodA()` calls `this.methodB()` and `methodB()` carries the `@Transactional` annotation that is failing, the proxy is being bypassed. The fix is to inject `self` or use `AopContext.currentProxy()`.

**Step 3 — Is the method public?**
Spring AOP (via CGLIB or JDK proxy) can only intercept public methods by default. A `@Transactional` annotation on a package-private, protected, or private method is silently ignored.

**Step 4 — Is the method final?**
If using CGLIB (which is the Spring Boot default), final methods cannot be overridden and therefore cannot be intercepted. I would remove the `final` modifier.

**Step 5 — Exception type and rollbackFor**
Spring rolls back only on `RuntimeException` and `Error` by default. If the method throws a checked exception and the developer expects a rollback, they need `@Transactional(rollbackFor = Exception.class)` or the specific checked exception class.

**Step 6 — Transaction propagation**
If two `@Transactional` methods interact and the inner one is configured with `Propagation.REQUIRES_NEW` but the call goes through `this` (self-invocation), the new transaction is never created. I trace the propagation chain to verify that separate transactions are actually created when expected.

**Step 7 — Transaction manager configuration**
In applications with multiple data sources, a method annotated `@Transactional` without specifying `transactionManager` uses the default one. If the method operates on a secondary data source managed by a different `PlatformTransactionManager`, no transaction is active for that data source.

After working through this checklist in roughly 5 minutes, most `@Transactional` failures are diagnosed. The most common culprit by far is self-invocation (Step 2), followed by incorrect exception type (Step 5).
